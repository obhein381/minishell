# Minishell

> A Unix Shell implementation project that mimics Bash behavior.

* **Language:** C
* **Environment:** Linux (Ubuntu)
* **Team Size:** 1
* **Development Period:** 2026.03.17 ~ 2026.05.18

This project implements a Unix shell that supports command execution, pipelines, redirections, environment variable expansion, heredoc, and signal handling.

# Motivation

I built this project to better understand how Bash works internally.

By implementing the entire execution flow—from tokenization and expansion to parsing and execution—I gained a deeper understanding of Unix processes, file descriptors, signals, and shell architecture.

# Build & Run

```bash
make
./minishell
```

# Features

This project was developed with a strong focus on Bash compatibility, including environment variable expansion, heredoc behavior, signal handling, and exit status propagation.

* Pipe (`|`)
* Input Redirection (`<`)
* Output Redirection (`>`)
* Append Redirection (`>>`)
* Heredoc (`<<`)
* Environment Variable Expansion (`$VAR`)
* Exit Status Expansion (`$?`)
* Built-in Commands

  * echo
  * cd
  * pwd
  * export
  * unset
  * env
  * exit
* Signal Handling (`SIGINT`, `SIGQUIT`)

# Usage Examples

minishell$ echo hello | cat

# Architecture Design

To separate responsibilities and reduce coupling, the project is divided into four stages:

* Lexer
* Expander
* Parser
* Executor

Each stage is responsible for a single concern:

* String interpretation
* Expansion
* Command structure generation
* Command execution

# Architecture

```text
User Input
    ↓
Lexer
    ↓
Token List
    ↓
Expander
    ↓
Expanded Tokens
    ↓
Parser
    ↓
Command List
    ↓
Executor
    ↓
Builtin / External Command
```

## Lexer

The Lexer converts the input string into tokens.

Operators that affect command structure, such as pipes (`|`) and redirections (`<`, `>`, `<<`, `>>`), are tokenized separately.

It also tracks quote states so that spaces inside quotes are not treated as argument separators.

## Expander

The Expander is responsible for:

* Environment variable expansion (`$VAR`)
* Exit status expansion (`$?`)
* Quote removal

The behavior follows Bash rules:

* No expansion is performed inside single quotes (`''`).
* Environment variables are expanded inside double quotes (`""`).

Since expansion results can affect command structure, expansion is performed before parsing.

## Parser

The Parser transforms tokens into executable command structures.

Responsibilities include:

* Pipe syntax validation
* Redirection syntax validation
* Command structure generation

The result is a linked-list based command structure containing:

* argv
* fd_in
* fd_out
* Redirection information

and other execution-related data.

## Executor

The Executor runs the command list generated by the Parser.

It first applies pipes and redirections, then executes commands.

Execution behavior varies depending on the command type.

### Built-in Commands

Commands that modify shell state:

* cd
* export
* unset
* exit

are executed in the parent process.

Executing them in a child process would not affect the shell's actual state.

### External Commands

External commands are executed in child processes created by `fork()`.

The shell resolves the executable path using `PATH` and executes it using `execve()`.

### Pipelines

For pipelines, all commands are executed in child processes, matching Bash behavior.

`pipe()` and `dup2()` are used to connect the standard input and output of each process.

# Troubleshooting

## Heredoc Signal Handling

### Problem

Pressing `Ctrl+C` during heredoc input should cancel only the heredoc operation.

However, the initial implementation sometimes terminated the entire shell or left it in an inconsistent state.

### Root Cause

The heredoc input was processed directly by the shell process.

As a result, a `SIGINT` generated during heredoc input was also delivered to the shell process itself.

### Solution

Heredoc input handling was moved to a dedicated child process.

* Parent Process: Maintains the shell
* Child Process: Handles heredoc input

This allows `SIGINT` to terminate only the heredoc process while keeping the shell alive.

### Result

* `Ctrl+C` cancels only the heredoc
* The shell continues running normally
* Behavior matches Bash

---

## Non-interactive Heredoc Handling

### Problem

After moving heredoc handling to a child process, interactive heredocs worked correctly.

However, heredocs behaved incorrectly when input was provided through a pipe:

```bash
printf "..." | ./minishell
```

### Root Cause

The implementation always processed heredoc input in a child process.

When using piped input, `get_next_line()` may prefetch multiple lines into its internal static buffer.

After `fork()`, the child process consumes heredoc input, but the parent's buffer state remains unchanged.

As a result, lines that should have been consumed by the heredoc were later interpreted as normal shell input.

### Solution

The shell now distinguishes input environments using:

```c
isatty(STDIN_FILENO)
```

* TTY environment → heredoc handled in a child process
* Non-TTY environment → heredoc handled in the parent process

The heredoc reading logic was also refactored into a dedicated `read_heredoc_to_pipe()` function.

### Result

* Interactive heredocs work correctly
* Non-interactive heredocs work correctly
* Heredoc content is no longer interpreted as subsequent commands

### Lessons Learned

Although both appear as standard input, terminal input and piped input behave differently.

This issue also highlighted how internal state (such as `get_next_line()`'s static buffer) interacts with process duplication through `fork()`.

---

## Exit Status Handling

### Problem

The value of `$?` sometimes differed from Bash behavior after executing external commands or pipelines.

Commands that completed successfully could be reported as failures.

### Root Cause

The shell waited for child processes using `waitpid()`, but the returned status was not fully interpreted.

In particular, the pipeline status conversion logic only considered signal-based termination.

```c
if (!WIFSIGNALED(status))
    return (1);
```

As a result, normally exited processes (`WIFEXITED`) always returned `1`.

### Solution

The status returned by `waitpid()` is now interpreted correctly.

```c
if (WIFEXITED(status))
    return (WEXITSTATUS(status));
```

For signal termination:

```c
if (WIFSIGNALED(status))
    return (128 + WTERMSIG(status));
```

This matches Bash's exit status rules.

### Result

```bash
true | cat
echo $?
```

Output:

```text
0
```

```bash
false | cat
echo $?
```

Output:

```text
1
```

Exit status propagation now behaves consistently with Bash.

### Lessons Learned

Executing a process and propagating its exit status are separate concerns.

Accurately reproducing Bash behavior requires careful handling of process termination semantics.
